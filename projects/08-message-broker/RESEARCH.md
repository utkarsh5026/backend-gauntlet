# Message Brokers in 2026: Internals, Architecture, and the Object-Storage Revolution — A Builder's Field Guide

## TL;DR
- **The log is the master abstraction.** Every high-throughput broker worth studying (Kafka, Pulsar, Redpanda, NATS JetStream, RabbitMQ Streams, Redis Streams, Fluvio, Iggy) is fundamentally an append-only, offset-addressed commit log; queues and pub/sub are views layered on top. If you build one thing well in Rust, build a correct, fast segmented log with sparse indexes.
- **The biggest active shift is "diskless"/object-storage-first design.** WarpStream, AutoMQ, Bufstream, Confluent Freight, and now the Apache Kafka project itself (KIP-1150 "Diskless Topics," accepted March 2026) are moving primary storage to S3 to eliminate cross-AZ replication traffic — which WarpStream's founders say is typically 80%+ of the cloud cost of running Kafka — accepting higher latency (hundreds of ms) in exchange for radically lower cost and stateless, elastic compute.
- **For a Rust builder, the design space is well-mapped:** thread-per-core + io_uring (Redpanda's C++/Seastar model, mirrored in Rust by Iggy on compio, and available via glommio/monoio) is the performance frontier; Raft (via a library like `raft-rs` or `openraft`) is the pragmatic replication choice; and the Kafka wire protocol is the de facto compatibility standard if you want adoption.

## Key Findings

1. **Delivery semantics are producer+broker+consumer properties, not a single switch.** At-most-once = fire-and-forget (no retries / commit-before-process). At-least-once = retries + ack-after-process (the pragmatic default; requires idempotent consumers). Exactly-once is achievable *within a system* via idempotent producers (dedup by identity+sequence number) plus transactions (atomic multi-partition commit), but is impossible to extend transparently to arbitrary external side-effects — the honest framing is "effectively once" via idempotent sinks.

2. **Kafka's design is the reference implementation of nearly every core technique** you'll want: segmented append-only log, sparse offset/time indexes, page-cache reliance, zero-copy `sendfile`, batching/compression, the ISR replication model with high-watermark, leader epochs to prevent split-brain, and (since KRaft) a Raft-based metadata plane.

3. **There is no free lunch across latency/throughput/cost/durability.** Redpanda optimizes tail latency (thread-per-core, no JVM, bypass page cache, `fsync`-per-write option); WarpStream optimizes cost/ops (S3-primary, stateless); Kafka optimizes ecosystem maturity and balanced throughput. Vendor benchmarks are "benchmarketing" — Redpanda's 10x claims were directly contested by Confluent's Jack Vanlightly on identical hardware.

4. **Rust is a first-class citizen in this space now.** Fluvio (Rust + WASM SmartModules) and Apache Iggy (incubating; thread-per-core + io_uring + compio) are the two most relevant native projects, and the async runtime ecosystem (tokio, glommio, monoio) directly supports both work-stealing and thread-per-core models.

## Details

### 1. Fundamentals and Core Abstractions

**The log/commit-log vs queue vs pub/sub.** Jay Kreps's 2013 essay "The Log: What every software engineer should know about real-time data's unifying abstraction" is the foundational text. A log is the simplest storage abstraction: an append-only, totally-ordered-by-time sequence of records, read left-to-right, each addressed by a monotonic offset. Kreps's key insight is that a log is simultaneously (a) a messaging system with durability and strong ordering ("atomic broadcast"), (b) a replication primitive (state-machine replication: feed the same log to deterministic replicas and they converge), and (c) a data-integration backbone. The original Kafka paper (Kreps, Narkhede, Rao, NetDB 2011) instantiates this.

Three abstractions sit on top of the log:
- **Log/stream** (Kafka, Pulsar, Redpanda, Redis Streams, RabbitMQ Streams): consumers track their own offset; messages are *not* deleted on read; multiple independent consumers replay at will; retention is time/size-based. Best for event sourcing, replay, high fan-out, stream processing.
- **Queue** (RabbitMQ classic/quorum, SQS): messages are consumed destructively (competing consumers), the broker tracks per-message delivery/ack state, and the queue trends toward empty. Best for work distribution/task processing.
- **Pub/sub** (SNS, core NATS, Google Pub/Sub): topic-based fan-out; each subscription is an independent delivery cursor. Can be ephemeral (core NATS "send-and-pray") or durable (JetStream, Pub/Sub).

**Delivery semantics — how they're actually implemented.**
- *At-most-once:* producer uses `acks=0` or doesn't retry; consumer commits offset *before* processing. Data loss on failure; zero duplication.
- *At-least-once:* producer retries on ambiguous failure; consumer commits *after* processing. The broker may write-then-fail-before-ack, so the producer resends → duplicates. This is the default in most systems.
- *Exactly-once (Kafka EOS, since 0.11):* two mechanisms combine. (1) **Idempotent producer**: the broker assigns each producer a **Producer ID (PID)**; each record batch carries the PID plus a per-partition monotonic **sequence number**; the broker tracks the last sequence per (PID, partition) and rejects duplicates (seq ≤ last) or gaps, persisting this state in a `.snapshot` file so it survives leader change. Since Kafka 3.0 this is on by default (`acks=all`, `enable.idempotence=true`), giving dedup + ordering. (2) **Transactions**: a `transactional.id` maps to a **transaction coordinator** that logs state in the internal `__transaction_state` compacted topic; the producer calls `initTransactions`/`beginTransaction`/`commitTransaction`; consumers set `isolation.level=read_committed` and read only up to the **Last Stable Offset (LSO)**, filtering aborted records. Limitations: one active transaction per producer (throughput ceiling), added latency from synchronous coordinator RPCs, and consumers need extra index files to filter aborted messages. Crucially, Kafka transactions cover *only* Kafka reads/writes+offset commits — not external DB writes; cross-system exactly-once requires the transactional read-process-write loop plus idempotent external sinks.

**Ordering guarantees.** Kafka/Redpanda/Kinesis guarantee ordering *per partition/shard only* — there is no global order across partitions (that would serialize the system). Ordering is tied to the partitioning key: same key → same partition → ordered. Redis Streams and Pulsar (single-partition topics) can give stronger per-topic ordering. Global total order requires a single partition (throughput bottleneck) or an external sequencer. Causal ordering is generally left to the application (via keys or vector clocks).

**Idempotency & deduplication** are implemented via (identity, sequence) pairs (Kafka PID+seq), content hashes / dedup IDs with a time window (SQS/SNS FIFO: 5-minute dedup window; NATS JetStream: `Nats-Msg-Id` + dedup window), or application-level idempotency keys.

**Push vs pull.** Kafka and Redpanda use **consumer pull** (`Fetch` requests): consumers control rate, batching is natural, and slow consumers can't be overwhelmed — this is also why Kafka's replication protocol is pull-based (followers fetch from the leader), which scales to many observers. RabbitMQ classically **pushes** to consumers (with a prefetch/QoS window for flow control). NATS core pushes; JetStream offers both push and pull consumers. Push gives lower latency for point-to-point; pull gives better backpressure and batching for high throughput.

**Backpressure & flow control.** Pull models get backpressure for free (consumer simply fetches slower). Push models need explicit windows: RabbitMQ uses consumer prefetch counts and TCP back-pressure + credit-based flow control internally; AMQP 1.0 has link credit. Producers experience backpressure via bounded in-flight requests / buffer memory (Kafka producer `buffer.memory`, `max.in.flight.requests.per.connection`). Broker-side: Kafka relies on the OS page cache + bounded request queues; Redpanda's Seastar has per-shard I/O queues with prioritization (latency-critical vs bulk).

**Acknowledgment protocols.** Producer→broker: `acks=0/1/all` trades durability for latency. Broker→consumer: offset commit (Kafka: auto or manual commit to `__consumer_offsets`), explicit ack (RabbitMQ `basic.ack`/`nack`, NATS `AckAck`, SQS delete-message), or cursor persistence (Pulsar stores cursors in BookKeeper; Redis Streams uses XACK + a Pending Entries List).

### 2. Deep Dive: Major Current Brokers

**Apache Kafka.**
- *Storage:* Each partition is a directory of **segment** files. The active segment is the only writable one; it rolls at `log.segment.bytes` (default 1 GB) or `log.roll.ms` (default 7 days). Each segment has: `.log` (record batches; filename = base offset, e.g. `00000000000000001007.log`), `.index` (sparse offset→byte-position map, one entry per `log.index.interval.bytes`, default 4096 B; 8 bytes/entry), `.timeindex` (timestamp→offset, 12 bytes/entry), `.snapshot` (producer sequence state), and `leader-epoch-checkpoint`. Lookups do a binary search on the sparse index then a short linear scan. Retention policies: `delete` (drop old segments) and `compact` (log compaction keeps the latest value per key — used by `__consumer_offsets` and `__transaction_state`).
- *Replication (ISR model):* Each partition has a leader and followers. Followers act as special consumers issuing `Fetch` requests. The **In-Sync Replica (ISR)** set contains replicas caught up within `replica.lag.time.max.ms` (default 30 seconds, raised from 16s between Kafka 2.4–2.5). Two pointers matter: **Log End Offset (LEO)** = last written offset; **High Watermark (HW)** = highest offset replicated to all ISR members. Consumers can only read up to the HW (prevents phantom reads of uncommitted data). A write with `acks=all` + `min.insync.replicas=2` is acknowledged only when all ISR members have it; if ISR shrinks below min, producers get `NotEnoughReplicas`. With f+1 replicas Kafka tolerates f failures (vs 2f+1 for majority-vote quorum) — the ISR approach needs fewer copies but waits for the *slowest* ISR member. **Leader epochs** (monotonic counter, bumped each election) prevent split-brain and log divergence: on recovery followers truncate to the leader-epoch's offset. `unclean.leader.election.enable=false` (default) prioritizes consistency (partition stays offline if no ISR member survives) over availability. KIP-966 (Eligible Leader Replicas) hardens the "last replica standing" case for deployments running without fsync.
- *Controller / KRaft:* Historically ZooKeeper stored metadata and the controller. **KIP-500** replaced this with **KRaft**: a Raft quorum of controller nodes stores all metadata in an internal single-partition `__cluster_metadata` log; brokers are *observers* that replicate this log and materialize it into an in-memory cache (event-sourced, periodically snapshotted per KIP-630). KIP-595 specifies the Raft variant — notably **pull-based** (replicas fetch, unlike push-based classic Raft) to reuse Kafka's log machinery and scale to many observers. Benefits: no ZK, faster failover (new controller already has state in memory), and scaling to millions of partitions (creating a 1000-partition topic dropped from ~10–15 s to near-instant). KRaft is production-ready since 3.3.1; **ZooKeeper was fully removed in Kafka 4.0 (2025)**.
- *Performance techniques:* **sequential I/O** (append-only turns random writes into linear ones), **page-cache reliance** (Kafka deliberately keeps little in JVM heap; hot data served from OS cache with zero disk reads for caught-up consumers), **zero-copy** via `sendfile`/`FileChannel.transferTo` (data goes page-cache → NIC without user-space copies; note: TLS and EOS defeat zero-copy because data must be processed in user space), **batching** (the "message set" abstraction amortizes network/disk overhead), and **compression** (gzip/snappy/lz4/zstd, best on larger batches). Replication factor 3 means 1 MB produced = ~3 MB disk + often 2× that in cross-broker network traffic.
- *Consumer groups & rebalancing:* Partitions are distributed across a group; a **group coordinator** (a broker) tracks membership via heartbeats. **Eager rebalancing** (original): stop-the-world — all consumers revoke all partitions, then reassign (Confluent measured 37,138 ms total pause in one test). **Cooperative/incremental rebalancing** (KIP-429, Kafka 2.4, `CooperativeStickyAssignor`): only partitions that must move are revoked, in two rebalance cycles; the same test dropped to 3,522 ms. **KIP-848** (Kafka 4.0) moves assignment logic broker-side (server-driven reconciliation), eliminating the group-wide synchronization barrier for faster, more stable rebalances. Static membership (KIP-345) avoids rebalances on rolling restarts.

**Apache Pulsar.**
- *Two-layer architecture:* stateless **brokers** (serving/compute) are fully decoupled from **Apache BookKeeper** "bookies" (storage). Brokers own no persistent data, so a broker can take over any topic instantly (no data movement) — a sharp contrast to Kafka, where a partition leader owns its data on local disk and rebalancing means copying gigabytes.
- *Storage model (segment-centric):* A topic is a **managed ledger** = an ordered list of **ledgers** (BookKeeper's basic storage unit, called *segments* in Pulsar terms). Only the final ledger is written to; sealed ledgers are immutable. Ledgers subdivide into **fragments**, the smallest unit of distribution across bookies. Each ledger writes to an **ensemble** of bookies with a configurable **write quorum** and **ack quorum** (e.g. ensemble 3, write 3, ack 2). Because data for one topic is striped across many bookies, a single topic can exceed any one node's disk — and you scale storage by adding bookies (new fragments land on them) independently from scaling brokers.
- *Split-brain avoidance:* BookKeeper **fencing** ensures only one writer (broker) can write to a ledger; on broker failure the new broker recovers the ledger by finding the last acknowledged entry across the ensemble, replicating forward, then sealing it. Because brokers are stateless, there's no split-brain to resolve at the compute layer.
- *Other:* first-class **multi-tenancy** (tenants/namespaces/topics), native **geo-replication**, **Pulsar Functions** (lightweight compute), **tiered storage** (offload sealed ledgers to S3/GCS, leveraging the segment-oriented, immutable design), and cursors (consumer positions) stored durably in BookKeeper. Metadata historically in ZooKeeper, now pluggable (Oxia, etcd).

**RabbitMQ.**
- *AMQP model:* producers publish to an **exchange**; **bindings** route (by routing key/pattern) to **queues**; consumers read from queues. Exchange types: direct, topic, fanout, headers.
- *Replication evolution:* **Classic mirrored queues** (leader + mirrors) were slow, unpredictable under partitions, and failed Jepsen testing — **removed entirely in RabbitMQ 4.0**. **Quorum queues** (since 3.8, built on the **Raft** consensus algorithm via the Ra library) are now the only replicated queue type: leader + followers, quorum writes, fast/clean leader election, no special catch-up process, with features like poison-message handling, at-least-once dead-lettering, and log compaction (Ra v3 minor/major compaction) to reclaim disk.
- *Streams:* since 3.9, an append-only, replicated log data structure (Kafka-like): non-destructive reads, replay from any offset, high fan-out. Complements queues (queues for work distribution, streams for event sourcing/audit). RabbitMQ 4.1+ adds SQL-like server-side filtering on streams for AMQP 1.0 clients.
- *Metadata store — Khepri:* Mnesia (the Erlang DB used since inception) is being replaced by **Khepri**, a Raft-based (Ra library) tree store. As of RabbitMQ 4.2/4.3, all replicated subsystems — quorum queues, streams, and Khepri — are Raft-based, making partition behavior far easier to reason about: a minority side of a split cannot make metadata changes (operations time out) rather than risking divergence. This is the same "pause_minority"-like behavior but enforced by Raft's design rather than configurable strategies.

**NATS / JetStream.**
- *Core NATS:* an extremely lightweight, in-memory pub/sub system with **subject-based routing** (dot-delimited hierarchies like `orders.us.east`, wildcards `*` single-token and `>` multi-token). At-most-once, "send-and-pray" — a message with no connected subscriber is lost. Ideal for RPC, telemetry, live dashboards.
- *JetStream:* the persistence layer built into the server binary. Adds **streams** (durable, replicated logs capturing subjects), durable/replayable **consumers** (push and pull), at-least-once (and dedup via `Nats-Msg-Id`), plus KV and object stores built on top. Clustering uses a **NATS-optimized Raft** — notably it combines the message-replication data plane with Raft consensus messages to reduce overhead, and uses **one Raft group per stream** (per-stream placement/replication flexibility at the cost of per-stream overhead). Data routing still flows over core NATS subject interest, so the metadata layer isn't in the hot path. Durability caveat worth copying: JetStream file streams flush to the OS but **fsync on a configurable `sync_interval` (default 2 minutes)** by default — so a single-node OS crash can lose recent data; replication is the durability mechanism.

**Redis Streams.**
- *Data structure:* an append-only log implemented as a **radix tree (Rax)** whose leaves are **listpack**-encoded "macro-nodes" packing many entries. Entry IDs are `<ms-timestamp>-<sequence>`, stored as 128-bit big-endian keys — because time-adjacent IDs share prefixes, the radix tree compresses them and supports O(log n) range/seek-by-ID queries. This is remarkably memory-efficient (antirez measured ~1M messages in ~16 MB vs ~220 MB for a sorted-set+hash equivalent).
- *Consumer groups:* each group tracks a **Pending Entries List (PEL)** — a radix tree mapping delivered-but-unacked entry IDs to (consumer, delivery-time, delivery-count). `XREADGROUP` delivers and adds to PEL; `XACK` removes; `XCLAIM`/`XAUTOCLAIM` reassign stuck messages. Caveats for a builder: `XDEL` causes "Swiss-cheese" fragmentation (memory freed only when a whole macro-node empties); prefer approximate trimming `XADD ... MAXLEN ~` which trims whole nodes; a stream lives on one shard (hot-spot risk at scale).

**Cloud-native brokers.**
- *Amazon Kinesis Data Streams:* shard-based, directly analogous to Kafka partitions. Each **shard** = 1 MB/s or 1,000 records/s in, 2 MB/s out (shared), max record 1 MB. Partition key is MD5-hashed to a 128-bit range to pick a shard; ordering + unique **sequence numbers** are per-shard. Scaling is via **shard split/merge** (parent shards go CLOSED); order across resharding preserved via KCL (state in DynamoDB). Retention default 24 h, extendable to 7 days then 365 days. **On-demand** (auto-managed) vs **provisioned** (you set shard count) modes. **Enhanced Fan-Out** gives each of up to 20 registered consumers a dedicated 2 MB/s/shard pipe over an HTTP/2 push API (`SubscribeToShard`, ~70 ms latency vs ~200 ms polling).
- *Google Cloud Pub/Sub:* cleanly separates control plane (**routers**, using a consistent-hashing variant) from data plane (**forwarders**, any of which can serve any topic/subscription — so the control plane upgrades without disrupting clients). On publish, a forwarder writes to a majority of N (odd) clusters, each to a majority of M (odd) disks, before acking. It is **partitionless / auto-scaling** with per-message parallelism — Google explicitly contrasts this with partition-bound systems that cap concurrency at the partition count. Default **at-least-once**; opt-in **exactly-once** (pull subscriptions only, within one region). Ack deadline default 10 s (range 10–600 s). **Ordering keys** give per-key order within a region (per-key publish capped at 1 MB/s). **Pub/Sub Lite** (the partition-based cheaper variant) is deprecated/end-of-life as of 2026.
- *AWS SQS/SNS:* SQS **standard** queues = at-least-once, best-effort ordering, near-unlimited throughput, stored redundantly across multiple servers. SQS **FIFO** = exactly-once processing + strict order within a `MessageGroupId`, 5-minute dedup window, 300 TPS (3,000 batched; up to 70,000 in high-throughput mode). **Visibility timeout** (default 30 s, 0–12 h) hides an in-flight message until deleted or timeout; **DLQ** captures messages exceeding `maxReceiveCount`; retention default 4 days (1 min–14 days). SNS is pub/sub fan-out (a message published once is replicated to all subscriptions); **standard vs FIFO topics** mirror SQS semantics; SNS FIFO topics can only deliver to SQS queues. The canonical durable fan-out pattern is SNS → multiple SQS queues each with its own DLQ.

### 3. Newer / Innovative Systems — What's Happening Now

**The diskless / object-storage-first shift.** Traditional Kafka was designed for 2011 data centers where cross-machine networking was effectively free and local disks were cheap. In the cloud both assumptions inverted: AWS charges $0.01/GB in each direction for cross-AZ transfer within a Region — i.e. $0.02/GB round-trip, and (as Corey Quinn of *Last Week in AWS* notes) each cross-AZ gigabyte is billed as 2 GB — while replicated local SSD (EBS/NVMe) costs ~$0.39/GiB/month vs S3 at ~$0.021/GiB/month (roughly 18–24× cheaper). WarpStream co-founders Richard Artoul and Ryan Worl frame the core problem bluntly: more than 80% of Kafka cloud costs were driven by inter-AZ networking fees — a structural problem that could only be solved by eliminating disk-based replication entirely — because Kafka replicates every write across AZs and producers often write cross-zone.

- **WarpStream** (founded 2023 by Richard Artoul & Ryan Worl; acquired by Confluent on Sept 9, 2024; Confluent in turn became an IBM subsidiary when IBM completed its ~$11 billion all-cash acquisition on March 17, 2026): replaces stateful brokers with stateless Go **Agents** (no JVM, no local disk) that speak the Kafka protocol and write **directly to object storage**. Any Agent can serve any topic, commit any consumer group offset, or act as coordinator — so autoscaling is trivial (like running nginx) and there are no leader elections or partition rebalances (consensus is offloaded to WarpStream's cloud metadata store / control plane). The tradeoff is latency: baseline S3 gives hundreds of ms; using **S3 Express One Zone** cuts end-to-end latency ~4× — WarpStream states "Lightning Topics deliver 33ms median produce latency on S3 Express One Zone," with writes quorum'd across multiple directory buckets to survive AZ loss. Data stays in the customer's own S3 buckets in standard format (no lock-in).
- **AutoMQ:** Kafka-compatible via **S3Stream**, which reimplements Kafka's `UnifiedLog`/`LocalLog`/`LogSegment` to store on S3. It keeps stateful-looking brokers but makes them **stateless** by writing first to a low-latency **WAL** (EBS, Regional EBS, or S3 WAL) for fast durable ack, then asynchronously uploading to S3 as primary storage; hot reads from an in-memory cache, catch-up reads from a separate cold cache to isolate backfill from real-time. Claims P99 < 10 ms with the right WAL. Uses EBS multi-attach for WAL recovery on broker failure. Contrast with WarpStream: AutoMQ has a WAL (lower latency, but a local-storage dependency in the write path); WarpStream is pure diskless (higher latency, zero local storage).
- **Redpanda:** not diskless — the opposite bet. A **C++ rewrite** of the Kafka protocol on the **Seastar** framework (from ScyllaDB), using **thread-per-core / shared-nothing**: each core is a "shard" with pinned memory (NUMA-local), its own I/O and task queues, and single-producer/single-consumer message queues forming an all-to-all mesh — no locks on the hot path, no cross-core data sharing. It **bypasses the Linux page cache** (Redpanda understands its own access patterns better than the OS) and manages its own memory/DMA I/O, uses `io_uring`, `O_DIRECT`, and (26.1+) profile-guided optimization. Each partition is a Raft group; ~1 GB/s writes per core, ~2 cores to saturate an NVMe drive. Newer additions: write caching (24.1, trades durability for up to 90% lower latency) and tiered storage.
- **Bufstream** (by Buf): self-hosted, drop-in Kafka replacement writing to S3, with strong Protobuf/schema governance; AWS + GCP only.
- **Confluent Freight:** Confluent Cloud's diskless-style, cost-optimized cluster tier.

**Kafka's own diskless future (KIP-1150).** The Apache Kafka community accepted **KIP-1150 "Diskless Topics" on March 2, 2026**. As Aiven's Josep Prat recorded, "On March 2, 2026, the vote to accept Diskless topics into Apache Kafka passed with overwhelming support of 9 binding votes and 5 non-binding ones"; the KIP was authored by Greg Harris, Ivan Yurchenko, Jorge Quilcate and others (Apache Kafka wiki; JIRA KAFKA-19161). This validates the WarpStream/AutoMQ direction inside Kafka itself. As analyzed by Jack Vanlightly, revision 1 uses a **leaderless** design: any broker serves any partition; brokers batch and upload **Shared Log Segment Objects (SLSOs)** to S3; **Batch Coordinators** (replicated state machines) sequence batches and assign offsets for per-partition global order. Notably, three competing KIPs (**KIP-1150, KIP-1176, KIP-1183**) simultaneously address cross-AZ replication cost — a genuine fork-in-the-road for the project; implementation has since been split into KIP-1163 (Diskless Core) and KIP-1164 (Coordinator), in active design as of April 2026. The "diskless" topics sit alongside classic disk-backed topics (which still handle transactions, compaction, and Kafka Streams state stores in early revisions).

**Thread-per-core / shared-nothing & modern async I/O.** This is the dominant performance paradigm for new brokers. Seastar (C++) pioneered it for ScyllaDB/Redpanda; the Rust ecosystem now offers equivalents. **io_uring** (Linux 5.1+) is central: unlike epoll's readiness model (which still requires a separate syscall to do the I/O, i.e. two user↔kernel transitions per op), io_uring is **completion-based** — submit batches of operations, the kernel performs them and posts completions (one transition per batch). This suits a log broker's sequential-append pattern perfectly.

**Stream-processing convergence & "streaming databases."** Kafka Streams and Flink increasingly blur the broker/processor line; WarpStream argues partitions are too low-level an abstraction for stream apps; Fluvio embeds WASM SmartModules for inline transformation (eliminating a separate Flink/Kafka-Streams cluster). Tableflow/Iceberg integrations materialize topics as queryable tables.

### 4. Internals Relevant to BUILDING One (Most Important)

**Storage engine.** Copy Kafka's proven design: per-partition directory of append-only segment files; filename = base offset; roll on size/time; a **sparse offset index** (offset→byte position, one entry per N bytes) plus a **sparse time index** (timestamp→offset); binary-search the index then linear-scan. Keep the active segment memory-mapped or use buffered append. Support two retention modes (delete by time/size; compaction keeping latest-per-key). Iggy — a Rust broker — follows exactly this: append-only `.log` files with `.index` files, segments sealed at 1 GiB, vectored writes through io_uring.

**Achieving high throughput.** The non-negotiables: (1) **sequential I/O** everywhere — append-only, no in-place updates; (2) **batching** — a message-set/record-batch abstraction shared byte-for-byte across producer/broker/consumer so you can transfer without re-encoding; (3) **avoid fsync-per-message** — rely on replication for durability and/or **group commit** (batch many appends per fsync); Kafka famously does *not* fsync per write and instead trusts ISR replication (fsync-per-write can cost 2–3 orders of magnitude in throughput). (4) **zero-copy** via `sendfile` for the consume path (defeated by TLS/encryption and by any per-record transformation); (5) **page cache** — let the OS cache hot data (Kafka) *or* manage your own cache and use O_DIRECT (Redpanda) — pick one deliberately; (6) **compression** on batches.

**Replication protocol — the central design choice.** Two families:
- **ISR/primary-backup (Kafka):** leader + dynamically-maintained in-sync set; commit when all ISR ack; tolerate f failures with f+1 replicas; needs a separate metadata/consensus layer (KRaft) for leader election and ISR membership. Lower replica count, but the metadata plane is a separate consensus system.
- **Quorum consensus (Raft/Paxos):** each partition/queue is its own replicated state machine (Redpanda: per-partition Raft; RabbitMQ quorum queues: per-queue Raft; NATS JetStream: per-stream Raft; Kafka KRaft: Raft for metadata only). Raft gives you leader election, log replication, and membership changes in one well-understood package; tolerate f failures with 2f+1 nodes. For a from-scratch Rust broker, **per-partition Raft via `openraft` or `raft-rs` is the pragmatic path** — it folds replication + leader election + split-brain avoidance into one algorithm, avoiding the need to build a separate ISR+controller system.

**Consensus & split-brain.** Whatever you choose, you need: monotonic **terms/epochs** (Kafka leader epochs, Raft terms) so stale leaders are fenced; **log truncation/reconciliation** on rejoin (followers truncate to the divergence point); and a rule that a leader must host the complete committed log (Kafka's ISR-completeness invariant; Raft's leader-completeness property). Pulsar/BookKeeper's alternative — **fencing** at the storage layer with stateless compute — is elegant if you separate storage and serving.

**Partitioning/sharding.** Hash the key (Kafka: murmur2; Kinesis: MD5) to pick a partition; ordering is per-partition; partition count bounds consumer parallelism. Decide early whether partition count is fixed (Kafka — can only increase) or elastic (WarpStream/Pub/Sub — no user-visible partitions). In a thread-per-core design, map each partition to exactly one shard/core (Redpanda, Iggy) so all its I/O is core-local and lock-free.

**Wire protocol / framing.** Either implement the **Kafka protocol** (huge win: every existing client/connector works — this is the de facto standard, and the entire diskless ecosystem chose compatibility over novelty) or design a custom binary protocol (Iggy supports TCP custom-binary + QUIC + WebSocket + HTTP; NATS has a tiny text-ish protocol). Key framing concerns: length-prefixed frames, a record-batch format with CRC32 for corruption detection (truncate to last valid offset on mismatch), API versioning, and correlation IDs for pipelining.

**Consumer offset management.** Either broker-managed (Kafka's `__consumer_offsets` compacted topic; keyed by group+partition), storage-managed cursors (Pulsar in BookKeeper), or PEL-style per-message ack tracking (Redis, RabbitMQ, SQS). For a log broker, the compacted-offsets-topic approach is simple and replay-friendly.

**Idempotent producers.** Assign a producer ID, track per-(producer, partition) sequence numbers, reject duplicates/gaps, snapshot the sequence state so it survives leader change. This is a modest amount of code and delivers dedup + ordering — worth doing early.

### 5. Performance, Benchmarks, and Tradeoffs

The unavoidable tension: you cannot simultaneously optimize latency, throughput, cost, durability, and availability (the streaming analog of CAP). Concrete positions:
- **Redpanda:** lowest, most stable tail latency (thread-per-core, no GC, optional fsync-per-write). Best for real-time bidding, market data, fraud paths.
- **Kafka:** best ecosystem, balanced sustained throughput, familiar failure modes. The safe default.
- **WarpStream/AutoMQ/Freight:** best cost + elasticity for high-retention, spiky, or multi-AZ workloads; higher latency (WarpStream hundreds of ms baseline, ~33 ms on S3 Express; AutoMQ P99 < 10 ms with EBS WAL).
- **RabbitMQ:** lowest latency for point-to-point/task queues (push model), richest routing; lower peak throughput.

**On benchmark provenance — read this skeptically.** Redpanda's headline claim is "10x faster tail latencies with up to 3x fewer nodes" using a modified OpenMessaging Benchmark (Kafka 3.2.0 vs Redpanda 22.2.2, m5n.8xlarge clients). Confluent's Jack Vanlightly reproduced it on identical hardware (3× i3en.6xlarge) and concluded "Redpanda's claims regarding Kafka and even their own performance are greatly exaggerated" — finding Redpanda "unable to reach 1000 MB/s with TLS" and its advantage collapsing under minor workload changes (e.g. throughput dropping to ~850 MB/s when producers rose from 4 to 50), struggling at its data-retention limit and to drain large backlogs, while Kafka could push NVMe to its ~2 GB/s limit at `acks=1` where Redpanda could not (jack-vanlightly.com, May 15 2023). The honest takeaway: **vendor benchmarks reveal *intent*, not universal truth; always benchmark your own workload.** Independent academic runs (Baylor, NTU 2023) using OpenMessaging exist and are more neutral starting points.

### 6. Rust-Specific Ecosystem

**Existing Rust brokers/projects:**
- **Fluvio** (InfinyOn): a Rust-native streaming platform, Kafka+Flink-like. Kubernetes-inspired declarative control plane (SPU = Stream Processing Unit). Its differentiator is **SmartModules** — WASM functions (filter/map/aggregate/array-map) compiled from Rust (and other LLVM languages) that run inline in a sandbox, eliminating a separate stream-processing cluster. Uses tokio, memory-mapped index files, zero-copy transfers. Idle footprint ~50 MB RAM (vs Kafka's ~1 GB + JVM warm-up), per InfinyOn's figures.
- **Apache Iggy** (incubating; created April 2023 by Piotr Gankiewicz, in the Apache Incubator since Feb 2025): "persistent message streaming at laser speed." Kafka-like data model (Stream → Topic → Partition → Segment). The 0.6.0 release (Dec 2025) was a **major architectural bet: rewritten from a tokio-based runtime to a completion-based io_uring model on compio**, with thread-per-core shared-nothing, NUMA-aware CPU pinning, custom zero-copy (de)serialization, and multi-transport (TCP custom-binary, QUIC, WebSocket, HTTP). Reports >5,000 MB/s throughput and sub-ms P99; early adopters report up to 20M msg/s over TCP. **Key gap: no clustering yet** — single-node only, with clustering planned via **Viewstamped Replication (VSR)**. This is the single most relevant reference implementation for your project.
- **Others:** RobustMQ (Rust, multi-protocol MQTT/Kafka/NATS, unified storage).

**Async runtimes — the core decision:**
- **tokio:** the dominant, mature, work-stealing multi-threaded runtime. Tasks must be `Send`/`Sync`; a global reactor. Best ecosystem. Iggy *left* it for io_uring; but you can also run tokio in a **thread-per-core** configuration (current-thread runtime pinned per core), which TechEmpower benchmarks show is ~1.5–2× faster than the multi-threaded runtime for even workloads.
- **glommio:** thread-per-core, **io_uring**-based, Linux-only (kernel 5.8+), Direct I/O first-class, "shares" API for I/O prioritization — explicitly recommended for databases/proxies/**brokers**. Inspired by Seastar. Copies into internal buffers (borrow-friendly API, small overhead).
- **monoio** (ByteDance): pure io_uring/epoll/kqueue thread-per-core runtime; ownership-transfer ("rent") buffer model required because the kernel needs stable buffers for io_uring; benchmarks ~2× tokio at 4 cores, ~3× at 16 cores for even workloads; can *lose* to tokio for unbalanced workloads (no work-stealing).
- **compio:** completion-based runtime Iggy adopted; also cross-platform (works where io_uring isn't available).

**The tradeoff to internalize:** thread-per-core + io_uring gives the best throughput/latency and lets you map partition→core for lock-free hot paths, but **work-stealing (tokio) tolerates unbalanced load better** — if some partitions are hot and others idle, a thread-per-core design can leave cores idle while one is saturated. For a broker where you can shard partitions across cores and load is reasonably balanced, thread-per-core wins; if load is highly skewed and unpredictable, tokio's work-stealing may serve you better. Note glommio/monoio/compio-uring are Linux-centric (io_uring); if you need Windows/macOS, tokio or compio's fallback drivers matter.

## Recommendations

**Staged build plan for a Rust broker:**

1. **Phase 1 — Single-node log done right.** Build the segmented append-only log: segment files named by base offset, sparse offset + time indexes, CRC32 per batch, retention (delete + compaction), a shared record-batch format used on disk and wire. Choose your I/O model now: default to **thread-per-core + io_uring (glommio or compio)** mapping one partition to one core, following Iggy's proven layout. Benchmark against Iggy as your reference. *Benchmark threshold to proceed:* sustained sequential write ≥ ~1 GB/s/core and stable P99.

2. **Phase 2 — Protocol + clients.** Decide compatibility strategy. If you want adoption and ecosystem, implement the **Kafka wire protocol** (the de facto standard the entire diskless generation chose). If you want to move fast and control UX, do a custom binary protocol like Iggy but plan for a Kafka-compat shim later. Add idempotent producers (PID + per-partition sequence + snapshot) early.

3. **Phase 3 — Replication via Raft.** Adopt **per-partition Raft** (`openraft` is the more modern, async-first choice; `raft-rs` is battle-tested from TiKV) rather than building a bespoke ISR+controller. This folds leader election, log replication, membership changes, and split-brain avoidance into one algorithm. This is the path Redpanda, NATS JetStream, and RabbitMQ quorum queues all took. Get consensus correct before optimizing.

4. **Phase 4 — Choose your durability/cost point.** Decide deliberately: (a) *local-disk + replication* (Kafka/Redpanda model — lowest latency, highest cross-AZ cost); (b) *WAL + object storage* (AutoMQ model — low latency with an S3 backend); or (c) *pure diskless/S3-primary* (WarpStream model — lowest cost, highest latency, stateless elastic compute). For a greenfield cloud broker in 2026, seriously consider **object-storage-first** — it's where Kafka itself is heading (KIP-1150). If you go this route, study the Batch-Coordinator/SLSO design and WarpStream's metadata-store-as-consensus approach.

5. **Phase 5 — Delivery semantics & consumer groups.** Add consumer groups with **cooperative/incremental rebalancing** (don't build eager stop-the-world; learn from KIP-429/KIP-848 and put assignment logic server-side). Offer at-least-once by default; add transactions/EOS only if your use cases demand atomic multi-partition writes.

**Decisions that would change the plan:**
- If **latency is paramount** (sub-ms P99, financial/real-time): go local-NVMe, thread-per-core, offer fsync-per-write or write-caching modes; skip diskless.
- If **cost/elasticity is paramount** (high retention, spiky, multi-AZ): go object-storage-first and accept hundreds-of-ms latency (or single-digit-ms with S3 Express One Zone / EBS WAL).
- If **load is highly skewed/unpredictable**: reconsider strict thread-per-core in favor of tokio work-stealing.
- If you need **cross-platform** (not just Linux): io_uring runtimes (glommio/monoio) are out; use tokio or compio with fallback drivers.

## Caveats
- **Vendor benchmarks are marketing.** Every headline throughput/latency number here (Redpanda 10x, WarpStream 33 ms, AutoMQ P99 <10 ms, Fluvio 1.5x/20-30x, Iggy 5 GB/s and 20M msg/s) comes from the vendor or its ecosystem and was measured on specific, favorable hardware/workloads. Confluent's independent-ish reproduction directly contested Redpanda's claims. Treat all as intent, not fact; benchmark your own workload.
- **"Exactly-once" is always scoped.** Kafka EOS, Pub/Sub exactly-once, and SQS/SNS FIFO all guarantee it only within their own system boundary (and often within one region). True end-to-end exactly-once across external systems is not achievable transparently — it requires idempotent sinks.
- **Fast-moving area.** KIP-1150 was accepted March 2026 and its design is still evolving across revisions (now split into KIP-1163/KIP-1164, with competing KIP-1176/KIP-1183); WarpStream is now under Confluent/IBM (IBM's Confluent acquisition closed March 2026); Pub/Sub Lite is being turned down in 2026; RabbitMQ's Khepri is stabilizing as default; Iggy is pre-clustering. Verify current status before committing.
- **Some sourcing is secondary.** Several internals (e.g., Iggy's exact runtime internals, some cloud limits) came from engineering blogs, Medium posts, and community wikis rather than primary docs; the most load-bearing facts (Kafka storage/ISR/KRaft, Pulsar/BookKeeper, WarpStream/AutoMQ architecture, Rust runtime models) are corroborated by official documentation or first-party engineering sources.
- **Not covered in depth:** ActiveMQ/Artemis, NServiceBus (a .NET messaging *framework* layered over transports like RabbitMQ/Azure Service Bus, not a broker itself), MQTT brokers, and Apache Pulsar's KoP/Kafka-on-Pulsar. These are relevant but secondary to a log-broker build.
